# 业务需求草案 (BRD) - v1.3.2a 赛事闭环与平台代制模式

> **文档性质**：从路线图规划阶段提炼的 v1.3.2a 版本专属核心矿石。
> **文档位置**：此文件用作 `/iteration-execution-pipeline-v2` 执行管线的**标准入口先决条件**，后续的 1a 业务死锁步骤将从此文件起跑。

## 1. 核心业务定位
此迭代为“社团挑战赛”玩法的 MVP 版本，其终极目的是**跑通拉新闭环和展现团长荣誉感**。
- **业务主视角聚焦**：跑通 C 端用户的体验全闭环（参赛-投票-分享），由平台充当代加工厂（S 端负责繁琐发赛配置）。
- **坚决剔除的范围**：不开发任何面向团长的移动端长串表单创赛工具，本期不在产品前端赋予 B 端发赛负担。

## 2. 必须实现的落地功能要点 (Scope)

### 2.1. 前置拦截与入社卡点 (用户拉新生命线)
- 完全封锁个人独立常规报名通道。
- 参与对应赛事的实质性操作（报名参赛、投稿作品、参与投票），**前置校验必须强制挂钩加入专属社团**。
- 探测到未入社用户欲发生上述交互时，前端必须立刻拦截，并提示其转化为社团人员方可继续（CC-5 需配合解决）。

### 2.2. 用户端 (C端) 互动业务底座
- **投稿与展示**：允许参与者上传符合要求的作品（图片/视频媒介），形成列表进行展销。
- **限定期限互动**：按固定的起止日期为卡表计算生命周期。进行中的状态允许点赞投票，截止日后全盘冻结。
- **主舞台品牌感**：详情主版面需以最直白手段展现大赛的品牌主办方（归由【某某团长/ XX社团】荣誉冠名主办）。

### 2.3. 管理与大荣誉闭环 (团长大爽点)
- **业务定位：自动化荣誉分发**：确立“平台设置、系统执行、团长背书”的铁三角逻辑。
- **管理小抓手**：系统自动感知当前团长身份，在赛事主页内为其提供管理直通车。团长具备随时阅览、管理违规及查看自动化执行进度（CC-25）的特权。
- **社交核武器——电子大奖状**：系统定时结账并落定总榜单。不涉及真实物理奖品分配，但必须通过系统全自动生成包装精良、大字号的“电子奖杯图片”与荣誉喜报，里面必须内嵌清晰的团长颁发背书和公章印记。
- **自动化生命周期**：确保 B/C 两端感知的一致性实。到点即止，通过系统硬拦截维护赛事公平性与严肃性。

### 2.4. 平台端 (S端) 代办兜底层
- 上述流程的核心规则阀门（命名设置、收件/截票日期、奖励规则明细），抛开客户端，全部由业务线运营人员在 Web 管理后台针对单一社团定向创建与开通。
